# /pr-review

지정된 GitHub Pull Request(PR)의 변경 사항을 읽어와, GitHub 서버에 직접 개별 라인 단위로 코드 리뷰 코멘트를 등록하고, 전체 리뷰 내용을 요약한 Artifact 리포트를 대화창에 제공하는 스킬입니다.

---

## 1. PR 번호 입력 받기

실행 시 에이전트는 사용자에게 "어떤 PR을 리뷰할 것인지(예: #15)" 묻습니다.
사용자가 PR 번호를 알려주면 다음 단계로 넘어갑니다.

*(만약 사용자가 처음부터 `/pr-review #15`처럼 번호를 명시했다면 이 단계를 생략합니다.)*

## 2. PR 분석 및 GitHub 라인별 코멘트 등록

에이전트는 해당 PR의 변경 사항(diff)을 가져와 분석한 뒤, GitHub MCP 도구를 사용하여 GitHub 서버에 **개별 파일 라인 단위로 정확한 코드 리뷰 코멘트를 직접 등록**합니다.

- **도구 사용**: `mcp_github-mcp-server_pull_request_read` (get_diff)로 변경점 및 라인 번호 파악
- **리뷰 등록**: `mcp_github-mcp-server_pull_request_review_write` 및 `add_comment_to_pending_review`를 통해 해당 파일의 특정 라인에 직접 리뷰 코멘트들을 생성한 뒤 일괄 제출(`submit_pending`)합니다.

## 3. 전체 리뷰 요약 Artifact 생성

GitHub에 개별 코멘트를 모두 달고 나면, 작성한 전체 코멘트 내용을 종합하여 **새로운 Artifact 문서(예: `pr-review-summary-N.md`)로 요약 리포트를 생성**합니다.

요약 리포트(Artifact)는 TDD(Red - Green - Refactor)의 사이클처럼, 전체 코드 리뷰의 흐름과 우선순위가 명확히 보이도록 **반드시 고쳐야 하는 사항부터 권장사항 순으로 나열**합니다.

### 요약 리포트 구조
1. **🔴 Must Fix (반드시 고쳐야 하는 치명적 결함)**
   - 로직 버그, 런타임 에러 가능성
   - 스펙(`SPEC.md`) 요구사항 누락 및 충돌
   - 심각한 성능 저하 및 보안 취약점
2. **🟡 Should Fix (수정을 강하게 권장하는 사항)**
   - 프로젝트 컨벤션(`GEMINI.md`, `DESIGN-supabase.md`) 위반
   - 타입 안정성 부족, 예외 처리 누락
3. **🟢 Refactor (더 나은 코드를 위한 리팩토링 제안)**
   - 가독성 향상, 불필요한 코드 중복 제거
   - 더 깔끔한 React/Next.js 패턴 적용
4. **✨ Good Points (칭찬할 점)**
   - 잘 작성된 로직이나 모범적인 코드

## 4. 후속 조치(피드백) 묻기

리뷰 리포트(Artifact) 작성을 마친 후, 에이전트는 사용자에게 이후 진행 방향에 대해 묻습니다.

> "리뷰가 완료되었습니다. 식별된 수정 사항들에 대해 어떻게 처리하시겠습니까?
> 
> 1. **지금 바로 고쳐야 하는 항목 이슈 생성**: 당장 수정이 필요한 치명적인 항목(Must Fix)들만 추려서 바로 GitHub 이슈로 등록합니다.
> 2. **중요도에 따른 분류 후 이슈 생성**: 리뷰에서 도출된 전체 작업들을 중요도(Must/Should/Refactor)에 따라 분류하여, 각각 성격에 맞게 분리된 GitHub 이슈들로 등록합니다.
> 
> 번호를 선택해 주시면 제가 이슈를 생성해 드리겠습니다!"

사용자의 선택에 따라 리뷰 내용을 바탕으로 GitHub 이슈를 생성(`mcp_github-mcp-server_issue_write` 또는 `gh issue create`)한 뒤 워크플로우를 종료합니다.

### 생성할 이슈의 형식 규약
이슈를 생성할 때는 다음 규약을 반드시 따릅니다.

1. **제목 (Title)**
   - 기존 이슈들의 네이밍 컨벤션에 맞춰 `[Prefix] [우선순위] 작업 요약` 형태로 작성합니다.
   - 예시: `[fix] [P0] API 보안 및 예외 처리 강화`, `[polish] [P2] 프론트엔드 성능 개선 및 접근성 제안`
   - Prefix는 `feat`, `fix`, `refactor`, `polish`, `chore` 등 작업 성격에 맞게 선택합니다.

2. **본문 (Body)**
   - 작업 목표와 구체적인 체크리스트 형태로 작성하여, 진행 상황을 트래킹할 수 있게 합니다.
   - 각 체크 항목에는 대상 파일 경로와 GitHub PR 리뷰 코멘트 원본 링크를 포함해야 합니다.
   - 예시 포맷:
     ```markdown
     ## 작업 목표
     PR #15 에 달린 코드 리뷰 코멘트를 반영하여 API 보안 및 예외 처리를 강화한다.

     ## 체크리스트
     - [ ] **존재하지 않는 ID 요청 시 null 200 반환 수정** ([리뷰 코멘트](https://github.com/.../pull/15#discussion_r...))
       - 경로: `app/api/tasks/[id]/route.ts`
       - 내용: 매칭되는 행이 없을 때 404를 반환하도록 수정
     ```
